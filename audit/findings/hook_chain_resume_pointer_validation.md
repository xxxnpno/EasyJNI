# hook_chain_resume_pointer_validation

## Summary
The `midi2i_hook` constructor now re-validates `chain_resume` with `is_valid_pointer` before baking it into the trampoline's resume JMP, and a failed check substitutes `nullptr` so the downstream `effective_resume` computation falls through to the default `target + HOOK_SIZE` resume (vmhook.hpp:5251-5253, 5484-5488). The hardening is correct and idempotent with the existing call-site filter inside `vmhook::hook<T>()`, but the constructor still silently truncates an out-of-32-bit-range `chain_resume` to a wrong `rel32` (no range check after `is_valid_pointer`), no unit coverage exists for the new defence-in-depth path, and the trampoline never re-checks `current_chain_resume` for staleness in `verify_and_repair`.

## Bugs

### [high] chain_resume that passes is_valid_pointer but is outside 32-bit JMP range silently truncates to a wrong rel32
- **file_lines:** vmhook/ext/vmhook/vmhook.hpp:5484-5492
- **description:** `is_valid_pointer` only checks canonical-address range, alignment and debug-poison sentinels; it does NOT enforce that the candidate sits within +/-2 GiB of `this->allocated + HOOK_SIZE + RESUME_JMP_OFFSET + RESUME_JMP_SIZE`. The trampoline buffer is constrained by `allocate_nearby_memory` to live in 32-bit JMP range of `target`, but `chain_resume` can be any user-space pointer the caller hands in (or any prior trampoline `vmhook::hook<T>()` discovered far away in the address space - the chain detector at 7878-7891 also only gates on `is_valid_pointer`). The `resume_jmp_delta = static_cast<std::int32_t>(ptrdiff)` silently truncates when the diff exceeds `INT32_MAX`, leaving a corrupt 5-byte JMP at `allocated + HOOK_SIZE + RESUME_JMP_OFFSET`; the very first non-cancel detour return jumps to a wild address and crashes the JVM. The defence-in-depth re-validation introduced in Unreleased does not catch this because the check is purely range/sentinel-based.
- **repro:** Call `new midi2i_hook(target, detour, chain_resume)` from a direct API consumer (or stage a chained prior trampoline) with `chain_resume = target + 0x9000'0000` (3 GiB away, still inside user-space canonical range and 8-byte aligned). `is_valid_pointer` returns true, `current_chain_resume` is set, `resume_jmp_delta` truncates to a small negative number and the trampoline JMPs into the middle of its own assembly on the first detour-resume.
- **suggested_fix:** After the `is_valid_pointer` gate, also verify reachability before baking the delta - either fail closed by setting `current_chain_resume = nullptr` (same fall-through to default) or by leaving the hook in its `error{true}` state. Concretely, after computing `effective_resume`, compute the `ptrdiff_t` first and check it fits in `std::int32_t` (`diff >= INT32_MIN && diff <= INT32_MAX`); if not, fall through to `target + HOOK_SIZE` (which is by construction reachable, since `allocate_nearby_memory` guarantees it) and log a warning. The same check should also be added inside `rewrite_chain_resume` (vmhook.hpp:5668-5669) where the recovery path computes the new delta with the same naked cast.
- **confidence:** certain

### [medium] verify_and_repair never re-validates a cached current_chain_resume; a freed/unloaded prior trampoline becomes a stale JMP target
- **file_lines:** vmhook/ext/vmhook/vmhook.hpp:5596-5616
- **description:** The Unreleased fix only re-validates `chain_resume` once, in the constructor. If the cached `current_chain_resume` initially passed `is_valid_pointer` but its owning DLL was later unloaded (or its trampoline allocation released), the next `verify_and_repair` round still happily preserves `new_chain = this->current_chain_resume` whenever `this->target[0]` is no longer `JMP_OPCODE`. The next time our detour returns "don't cancel", the trampoline jumps into freed/unmapped memory. The repair flow should re-run `is_valid_pointer` (or even `is_readable_pointer`) on the cached pointer before re-using it.
- **repro:** Install vmhook into a process that also has a prior i2i-patching DLL, then unload that DLL. Trigger `vmhook::verify_hooks()` after a benign third-party stomp of our JMP and observe the recovery path retain `current_chain_resume` pointing into the unloaded DLL's image. Next interpreted call into the patched method JMPs into freed memory.
- **suggested_fix:** At line 5596 replace `void* new_chain{ this->current_chain_resume };` with a re-validated default: `void* new_chain{ (this->current_chain_resume && vmhook::hotspot::is_valid_pointer(this->current_chain_resume)) ? this->current_chain_resume : nullptr };`. When `new_chain` is `nullptr`, also rewrite the trampoline's resume JMP back to `target + HOOK_SIZE` (the existing `rewrite_chain_resume` handles only the non-null path - it needs a small extension or a dedicated `rewrite_chain_resume_to_default()` helper).
- **confidence:** likely

### [low] const_cast on chain_resume drops the const qualifier without enforcing read-only intent
- **file_lines:** vmhook/ext/vmhook/vmhook.hpp:5236-5253
- **description:** The parameter is `const void* const chain_resume`, but it is stored after `const_cast<void*>(chain_resume)` into a non-const `current_chain_resume`. The trampoline only computes a delta from it and never writes through it, so the cast is semantically harmless, but the asymmetry tempts a future maintainer to mutate `current_chain_resume` via that field. Either drop the constness from the parameter (it is a code address that is never const-correct in practice) or store the field as `const void* const current_chain_resume` and `reinterpret_cast<const std::uint8_t*>` at the point of use.
- **repro:** Code-inspection only - no runtime behaviour today.
- **suggested_fix:** Declare the member as `const void* current_chain_resume{ nullptr };` and remove the `const_cast`. Update `rewrite_chain_resume` to also take a `const void*`. This costs nothing and removes a misleading const-strip.
- **confidence:** certain

## Improvements

### [S] [INTERNAL] Lift the `chain_resume && is_valid_pointer(chain_resume) ? ... : nullptr` ternary into a named static helper
- **rationale:** The exact same predicate is now duplicated at three sites: the constructor (5251), the `vmhook::hook<T>()` gateway (7883), and the verify_and_repair chain detector (5602-5603). Centralising it inside `midi2i_hook` (e.g. `static auto sanitize_chain_resume(const void*) noexcept -> const void*`) gives a single seam to extend later (the 32-bit-range check above, or an `is_readable_pointer` upgrade) and makes the intent self-documenting at every call site.
- **file_lines:** vmhook/ext/vmhook/vmhook.hpp:5251-5253, 5602-5606, 7878-7891
- **suggested_change:** Add a `private: static` member `sanitize_chain_resume(const void* p) noexcept -> const void* { return (p && vmhook::hotspot::is_valid_pointer(p)) ? p : nullptr; }`, then have all three sites call it. The constructor becomes `current_chain_resume{ const_cast<void*>(sanitize_chain_resume(chain_resume)) }` and `verify_and_repair` starts `new_chain` from `sanitize_chain_resume(this->current_chain_resume)`.

### [XS] [USER_FACING] Log a warning when a non-null chain_resume is rejected by the constructor
- **rationale:** Today a bad `chain_resume` is silently swallowed - the user passes a pointer, the trampoline silently uses a different resume target, and the log says nothing. The matching call site inside `vmhook::hook<T>()` at lines 7894-7899 already logs a `warning_tag` when its own `is_valid_pointer` filter rejects a candidate. The constructor's defence-in-depth should match: when `chain_resume != nullptr && !is_valid_pointer(chain_resume)`, emit a single `VMHOOK_LOG(warning_tag, ...)` so an end user debugging "why didn't my chain fire" can see the truncation in the log. Otherwise the new safety net is invisible.
- **file_lines:** vmhook/ext/vmhook/vmhook.hpp:5251-5253
- **suggested_change:** Replace the member-initialiser ternary with an inline lambda that logs on the rejection branch, or move the check into the constructor body just after the early `error` guard. Mirror the wording of the gateway warning at 7894-7899 (mention the failing pointer value and that the trampoline will fall back to `target + HOOK_SIZE`).

### [S] [INTERNAL] Tighten the documentation block to match the new defence-in-depth contract
- **rationale:** The doc comment for the `chain_resume` parameter (5223-5234) still reads as if the caller is fully trusted: "Defaults to `target + HOOK_SIZE`, but the install site can pass another DLL's trampoline entry here". It does not mention that anything failing `is_valid_pointer` is silently coerced to `nullptr`. A direct API consumer cannot tell from the doc whether their bad pointer will silently fall back or throw - especially confusing given the constructor doesn't throw on this path. Add one sentence to the `@param chain_resume` block: "Bad pointers (those that fail `vmhook::hotspot::is_valid_pointer`) are treated as `nullptr`; the trampoline falls back to the default `target + HOOK_SIZE` resume."
- **file_lines:** vmhook/ext/vmhook/vmhook.hpp:5223-5234, 5242-5250
- **suggested_change:** Add the sentence above to the `@param chain_resume` Doxygen block, and consolidate the inline comment at 5242-5250 to point at the doc instead of duplicating the rationale (or vice versa - keep one source of truth).

### [M] [INTERNAL] Hoist the WIN/SysV `RESUME_JMP_OFFSET` constants into a single struct-of-constants
- **rationale:** Three copies of the platform-conditional offsets (`HOOK_SIZE`, `JE_OFFSET`, `JE_SIZE`, `RESUME_OFFSET`, `RESUME_JMP_OFFSET`, `RESUME_JMP_SIZE`, `DETOUR_ADDRESS_OFFSET`) live in the constructor (5265-5380) and a redacted second copy lives in `rewrite_chain_resume` (5656-5663). The doc comment at 5647-5648 explicitly calls out the duplication risk ("MUST stay in sync with the assembly array in the ctor"). A `struct midi2i_trampoline_layout` with `inline static constexpr` members per ABI removes the duplication and lets a `static_assert(sizeof(assembly) == midi2i_trampoline_layout::TOTAL_SIZE)` catch any future edit-asymmetry at compile time. This is purely internal cleanup but makes the chain_resume rewrite path much harder to break.
- **file_lines:** vmhook/ext/vmhook/vmhook.hpp:5265-5380, 5650-5682
- **suggested_change:** Define a private nested `struct layout { static constexpr int HOOK_SIZE = 8; ... }` (with two specialisations behind `VMHOOK_OS_WINDOWS`) and replace every magic offset with `layout::FOO`. Add a `static_assert` after each `assembly[]` definition checking that `sizeof(assembly) == layout::ASSEMBLY_SIZE`.

## Parity Concerns
- The `vmhook::hook<T>()` gateway at vmhook.hpp:7878-7891 already gates `prior_trampoline` on `is_valid_pointer`. The constructor's defence-in-depth check is functionally identical, but the two sites are not factored into a shared helper - any future tightening (e.g. adding `is_readable_pointer` for VirtualQuery-style commit checking) has to be applied in two places, and someone will forget. See the [S] [INTERNAL] improvement above.
- The chain-resume rewrite path inside `verify_and_repair` (vmhook.hpp:5596-5616) is the THIRD copy of the same gate - it re-validates the *new* `prior_trampoline` discovered after a stomp, but unlike the constructor it never re-validates the *cached* `current_chain_resume` (see [medium] bug above). The defensive contract is inconsistent: install-time and recovery-time treat the cached value differently.
- `rewrite_chain_resume` (vmhook.hpp:5650-5682) performs the same naked `static_cast<std::int32_t>(ptrdiff)` as the constructor (vmhook.hpp:5490-5492) without any range check. If the high-severity 32-bit-range bug is fixed only in the constructor, the recovery path will still silently truncate. Fix both sites in the same change.
- The CHANGELOG entry promises "Bad input now falls through to the default `target + HOOK_SIZE` resume", but the constructor is silent on rejection (no log line). The matching warning at vmhook.hpp:7894-7899 in the gateway sets the user expectation that bad chain_resume IS observable in the log; the constructor should match. See the [XS] [USER_FACING] improvement above.
