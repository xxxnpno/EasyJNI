# hook_install_after_jit

## Summary
`vmhook::hook<T>()` installs a per-method deopt that overwrites `_from_interpreted_entry`, `_from_compiled_entry`, and clears `_code` when the snapshot at install time shows the method JIT-compiled. The install-time deopt catches *future* dispatch through `_from_compiled_entry`, but the README oversells it: direct compiled callers whose inline caches already cached the nmethod's verified-entry address still bypass the hook until a safepoint repairs the IC. The deopt write sequence also has a real-world race on ARM64 (no release ordering between the three pointer writes) and uses a stale snapshot (`was_compiled`) instead of re-reading `_code` at the deopt point, so an in-flight compilation that completes between snapshot and write is silently missed at install (the watchdog repairs it ~1 s later).

## Bugs

### [high] ARM64 store reordering between `_from_compiled_entry` write and `_code = nullptr` can hand HotSpot a stale-entry/null-code combination
- **file_lines:** vmhook/ext/vmhook/vmhook.hpp:7940-7947, vmhook/ext/vmhook/vmhook.hpp:6445-6447, vmhook/ext/vmhook/vmhook.hpp:2347-2356
- **description:** The install path does three plain pointer stores in order: `set_from_interpreted_entry(i2i)`, `set_from_compiled_entry(c2i_entry)`, `set_code(nullptr)`. The comment at line 7944 ("Clear `_code` last so the above entry-point writes are visible first") relies on store ordering, but `set_code` / `set_from_compiled_entry` / `set_from_interpreted_entry` (vmhook.hpp:2347, 2364, 2411) are plain non-atomic stores with no memory barriers. The header explicitly supports ARM64 (vmhook.hpp:176, `VMHOOK_ARCH_ARM64`). On ARM64 (and any other weakly-ordered arch) the CPU may commit `_code = nullptr` before the `_from_compiled_entry = c2i_entry` write becomes visible to another core, producing exactly the "`_code == nullptr` but `_from_compiled_entry` still points into the (now stale) nmethod — that combination crashes at the next safepoint" failure the install code already calls out at vmhook.hpp:7931-7932. The same hole exists in `deoptimize_methods_if` (vmhook.hpp:6445-6447), the verify_hooks repair path (vmhook.hpp:8268-8273), and the try_reinstall path (vmhook.hpp:8121-8125) — every per-method deopt site reads identically.
- **repro:** Run on an ARM64 host (Apple Silicon, Graviton) under load with a hot JIT'd method being hooked while another core is dispatching that method via its compiled entry. Use a `perf`/CoreSight stress harness to amplify reordering or run a chaos test that loops `hook` / `unhook` while the JIT thrashes recompiling. Crash will surface at the next safepoint in the snippet that reads `_code` and `_from_compiled_entry` together.
- **suggested_fix:** Make `set_code(nullptr)` a release store and the c2i-entry write either a release store or paired with a `std::atomic_thread_fence(std::memory_order_release)` before `set_code`. Concretely: change `set_code` (and ideally the entry-point setters) to use `reinterpret_cast<std::atomic<void*>*>(...)->store(code, std::memory_order_release)`, or insert `std::atomic_thread_fence(std::memory_order_release)` immediately before each `set_code(nullptr)` call site (lines 7945, 6447, 8125, 8273). On x86_64 this compiles to a plain `mov` (TSO already orders prior stores before later stores), so x86 perf is unchanged.
- **confidence:** likely

### [medium] `was_compiled` is captured before any synchronisation and the deopt branch re-uses the snapshot instead of re-reading `_code`, so an in-flight JIT completion after the snapshot is silently un-deopted
- **file_lines:** vmhook/ext/vmhook/vmhook.hpp:7790-7793, vmhook/ext/vmhook/vmhook.hpp:7933
- **description:** Line 7793 snapshots `was_compiled` from `original_code`, then the deopt branch at line 7933 keys off that snapshot. Setting `NO_COMPILE` at line 7781 *requests* the compiler to leave the method alone, but the comment at vmhook.hpp:8223-8227 says HotSpot "racing an in-flight compilation through the policy gate before it sees NO_COMPILE" is an observed source of `_code` re-population. So the window between `was_compiled = false` at 7793 and the missing deopt at 7933 is real: when an in-flight compile completes inside this window, install returns "interpreted" yet `_code` is non-null and `_from_compiled_entry` was never redirected. The watchdog (vmhook.hpp:8239-8273) does repair this ≤ `VMHOOK_AUTO_REPAIR_INTERVAL_MS` (default 1000 ms) later — but for that whole second every compiled caller calls the live nmethod and skips the hook. Install-path correctness shouldn't depend on a 1 s watchdog window.
- **repro:** Hook a method that's actively being tier-compiled (a Minecraft `runTick` or a tight `World.tick`). Timing-wise, the install path most often catches `_code == nullptr` because the snapshot happens before NO_COMPILE has propagated. Add a small sleep instrumentation in test harness between `set NO_COMPILE` and `original_code` snapshot, then between snapshot and deopt branch, and verify `_code` becoming non-null in the second window.
- **suggested_fix:** Re-read `_code` at the deopt site instead of using `was_compiled`: `void* const code_live{ found_method->get_code() }; if (code_live && vmhook::hotspot::is_valid_pointer(code_live)) { ... }`. Mirror the watchdog's repair logic so install and repair share the same path. Optionally loop while `_code` keeps flipping (bound the loop at ~3 iterations to avoid pathological JIT thrash). Same fix applies at vmhook.hpp:7795 for the log message, and the snapshot of `original_from_interpreted` / `original_from_compiled` (lines 7791-7792) can then be dropped entirely (see Improvements).
- **confidence:** likely

### [medium] README's "per-hook install already deoptimises the hooked method itself, which catches direct callers" overstates what the install does — stale inline caches still call the old nmethod directly
- **file_lines:** README.md:581-583, vmhook/ext/vmhook/vmhook.hpp:7919-7921
- **description:** The README at 581-583 promises "per-hook install … catches direct callers". The in-code comment at vmhook.hpp:7919-7921 says exactly the opposite: "Limitation: compiled callers with stale monomorphic inline caches still call the old nmethod directly. Those caches will be repaired the next time HotSpot reaches a safe point and re-evaluates the IC." So compiled callers with an already-resolved IC pointing at the nmethod's verified-entry-point skip `_from_compiled_entry` entirely — our c2i redirect only catches *re-resolutions* (megamorphic transitions, new IC misses, safepoint cleanups). The README phrasing leads users to skip `deoptimize_all_jit_compiled_methods()` thinking direct callers are already handled. This is a documentation bug, not a code bug, but the audit explicitly asked us to verify the claim.
- **repro:** Warm a tight Java loop that calls `target()` enough times for the caller to be C2-compiled and cache the monomorphic IC for `target`. Install a hook on `target` without calling `deoptimize_all_jit_compiled_methods`. Watch the hook fire 0 times for the next N caller iterations until a safepoint re-resolves the IC.
- **suggested_fix:** Rewrite README.md:581-583 to mirror the in-code comment, e.g. "vmhook's per-hook install marks the hooked method `_dont_inline`, clears `_code`, and redirects `_from_compiled_entry` to the c2i adapter — this catches every freshly-resolving compiled call site, but compiled callers whose inline caches already cached the original nmethod's entry will keep bypassing the hook until the next safepoint repairs the IC. Always pair installs against already-warm code with `vmhook::deoptimize_all_jit_compiled_methods()` (or a narrower predicate) to flush those stale ICs immediately."
- **confidence:** certain

### [medium] `vmhook::deoptimize_methods_if` snapshots `_code` once and then writes blindly — a concurrent re-JIT (or another agent's deopt) between read and write produces the same stale-entry/null-code crash mode the install path defends against
- **file_lines:** vmhook/ext/vmhook/vmhook.hpp:6417-6447
- **description:** The sweep at line 6417 reads `void* const code{ m->get_code() }` and treats a non-null result as "this method is JIT-compiled, deopt it." It then writes `set_from_interpreted_entry`, `set_from_compiled_entry`, `set_code(nullptr)` without re-reading `_code`. If `_code` flipped to nullptr between line 6418 and line 6447 (another agent's deopt, a HotSpot make-not-entrant during sweeper run), we're now writing into a method that's already in the deopted steady state — harmless but wasted writes. The more interesting direction is the reverse: if `_code` was null at line 6418 (we `continue` at 6420), but a tier-2 compile completes microseconds later, that method stays JIT-compiled and bypassed for the lifetime of `deoptimize_methods_if`. README documents this helper as the way to catch already-inlined callers, but it can race past a method whose nmethod was published microseconds late. Unlike the install path, there's no auto-repair watchdog for non-hooked methods, so the user has no recovery.
- **repro:** Run `deoptimize_all_jit_compiled_methods()` while a JIT compiler thread is publishing new nmethods (-XX:+PrintCompilation + a warming workload). Walk the deoptimisation log against PrintCompilation and look for methods that PrintCompilation shows as compiled at time T+ε but `deoptimize_methods_if` skipped at time T.
- **suggested_fix:** Re-read `_code` immediately before the write trio, and skip the writes if `_code` is now null: `void* const code_live{ m->get_code() }; if (!code_live || !is_valid_pointer(code_live)) continue;`. Document the snapshot semantics: "Iterates the class-loader-data graph as a single snapshot; a method that becomes JIT-compiled after the iteration cursor has moved past its class will not be touched." Optionally, accept a `repeat = N` parameter that loops the whole sweep up to N times to catch late-published nmethods.
- **confidence:** likely

### [low] `set_code(nullptr)` falls back to a silent no-op when the VMStructs `Method::_code` entry is missing — the install path then thinks deopt succeeded but `_code` was never cleared
- **file_lines:** vmhook/ext/vmhook/vmhook.hpp:2347-2357, vmhook/ext/vmhook/vmhook.hpp:7945
- **description:** `set_code` (vmhook.hpp:2347) early-returns silently if `iterate_struct_entries("Method", "_code")` returns null. The install path at line 7945 has no way to tell that the write was dropped — the success log at 7946-7947 prints regardless. On any JDK where the VMStructs export changes name (the codebase already handles the `_from_compiled_code_entry_point` → `_from_compiled_entry` rename for one field; if HotSpot ever does the same to `_code` we'll silently leak compiled bypasses with no diagnostic). Same hole in `set_from_interpreted_entry` (line 2364) and `set_from_compiled_entry` (line 2411).
- **repro:** Run vmhook against a JVM build where `Method::_code` has been stripped from VMStructs (e.g. a stripped-down embedded HotSpot). Install a hook on an already-JIT'd method. Observe the cheerful "deopt complete" log message at vmhook.hpp:7947 followed by the hook never firing for compiled callers.
- **suggested_fix:** Have `set_code` (and the entry-point setters) return `bool` indicating whether the write happened, and have the install path check the result and either fall back to the "c2i unrecoverable" branch (lines 7950-7967) or surface a clear error. At minimum, emit `VMHOOK_LOG(error)` from inside `set_code` on the fallback path so the diagnostic is visible.
- **confidence:** speculative

## Improvements

### [S] [INTERNAL] Drop dead `original_code`, `original_from_interpreted_entry`, `original_from_compiled_entry`, `was_compiled` from `hooked_method`
- **rationale:** All four fields are set at vmhook.hpp:7830-7836 and never read again after install. The shutdown comment at vmhook.hpp:7995-8007 explicitly explains why we *don't* restore `_code` (writing the saved value back to a freed nmethod was the post-uninject crash cascade). The `hooked_method` struct doc at vmhook.hpp:5693-5696 still claims they exist "so shutdown_hooks() can fully restore the method's state, including re-linking the nmethod" — stale by ~one release. Removing them shrinks every `hooked_method` by 32 bytes and deletes a stale doc comment.
- **file_lines:** vmhook/ext/vmhook/vmhook.hpp:5694-5705, vmhook/ext/vmhook/vmhook.hpp:7790-7836
- **suggested_change:** Delete `original_code`, `original_from_interpreted_entry`, `original_from_compiled_entry`, `was_compiled` from the struct (vmhook.hpp:5702-5705). Delete the matching snapshots in `hook<T>()` (vmhook.hpp:7790-7793 — keep a *local* `was_compiled` for the deopt branch or replace with the live re-read suggested in the bugs section). Delete the four arguments from the `hooked_method` aggregate-init at vmhook.hpp:7830-7836. Update the struct doc comment to say what shutdown actually does — leave the method in its install-time deopted state.

### [M] [USER_FACING] Make per-hook install accept an opt-in `deopt_callers` flag (default off) so users don't have to remember to call `deoptimize_all_jit_compiled_methods()` separately
- **rationale:** The README's two-step pattern ("install hook, then call `deoptimize_all_jit_compiled_methods`") is a footgun: forget the second call and your hook silently misses every already-warmed call site. Users who install hooks at injection time against an already-warm JVM almost always want both steps. Bundling the deopt sweep as an opt-in parameter on the hook itself (or a `vmhook::install_hooks(...)` batch wrapper) removes the discoverability problem and lets users be precise — "deopt only callers under `net/minecraft/`" — without writing the predicate lambda by hand.
- **file_lines:** vmhook/ext/vmhook/vmhook.hpp:7688-7711, vmhook/ext/vmhook/vmhook.hpp:6391-6478
- **suggested_change:** Add a struct `hook_options { std::optional<std::function<bool(const std::string&, vmhook::hotspot::method*)>> deopt_callers_matching; }` and an overload `vmhook::hook<T>(name, sig, user_detour, hook_options)`. After the existing install body, if `deopt_callers_matching` is set, call `deoptimize_methods_if(*opts.deopt_callers_matching)` once (or batch across multiple installs via a flush at the next safepoint). Document the cost (one heap scan) and that the predicate runs O(N_classes × N_methods).

### [S] [USER_FACING] Spell out the safepoint repair window in the "Catching already-inlined callers" README section so users understand *why* the deopt sweep is needed and *when* per-hook deopt alone is sufficient
- **rationale:** Today the README implies the only gap that needs the sweep is *inlined* call sites. The bigger gap is stale ICs in non-inlining callers — which a reader of the README has no way to know about. Three sentences fix it.
- **file_lines:** README.md:572-596
- **suggested_change:** Add: "vmhook's per-hook install marks the hooked method `_dont_inline`, clears `Method::_code`, and redirects `_from_compiled_entry` to the c2i adapter. That catches every fresh inline-cache resolution (and every interpreter dispatch). It does *not* invalidate inline caches that already cached the original nmethod's verified-entry-point — those compiled callers keep bypassing the hook until the next safepoint cleanup re-resolves the IC, typically tens of ms but unbounded in the worst case. If you need every call site routed through the hook immediately (rather than 'eventually'), call `deoptimize_all_jit_compiled_methods()` or a narrower predicate after install."

### [S] [INTERNAL] Tighten the "interpreted" log line at line 7801 — interpreted callers don't *always* hit the i2i stub patch, only after `_from_interpreted_entry` is verified to point at `_i2i_entry`
- **rationale:** The log line "is interpreted - patching i2i stub" implies the install will succeed for interpreted callers. If `_from_interpreted_entry` had been redirected elsewhere by another agent (Lunar, Forge mixin) the hook would silently not fire. The "was_compiled = false" branch never re-sets `_from_interpreted_entry`, so it's a real footgun if another agent has been there first.
- **file_lines:** vmhook/ext/vmhook/vmhook.hpp:7799-7802, vmhook/ext/vmhook/vmhook.hpp:7940-7946
- **suggested_change:** In the `else` branch (interpreted case), check whether `found_method->get_from_interpreted_entry() == i2i`. If not, either reset it (matching the compiled branch's `set_from_interpreted_entry(i2i)` at line 7941) or `VMHOOK_LOG(warning_tag, ...)` explaining that another agent has redirected the interpreted entry and the hook may not fire.

### [XS] [INTERNAL] `deoptimize_methods_if` final log line reports 0 totals when no methods were JIT-compiled, which looks like the helper "did nothing" — clarify
- **rationale:** Log line at vmhook.hpp:6451-6452 reads "deoptimize_methods_if: 0 deoptimised, 0 skipped (no recoverable c2i adapter)." for the entirely common case of "no methods matched the predicate." A user reading the log can't tell whether their predicate fired and matched nothing, or whether the predicate was never called. Add a third counter — `predicate_matched` or `predicate_examined` — so the log distinguishes "we walked 12 044 methods and your predicate matched 0" from "your predicate matched 250 but they were all interpreted-only."
- **file_lines:** vmhook/ext/vmhook/vmhook.hpp:6395-6452
- **suggested_change:** Add `std::size_t examined{ 0 }` (incremented in the inner loop), `std::size_t matched{ 0 }` (incremented after the predicate returns true), and include both in the log: `"deoptimize_methods_if: examined {} methods, matched predicate {}, deoptimised {}, skipped {} (no recoverable c2i adapter)."`

## Parity Concerns
- `vmhook::deoptimize_methods_if` and the per-hook install perform the same three-store deopt dance but diverge on logging detail and on the c2i-unrecoverable fallback policy: install (vmhook.hpp:7949-7967) clears `_code` anyway and falls back to interpreter-only correctness; the sweep (vmhook.hpp:6431-6437) bumps a `skipped_no_c2i` counter and leaves the method alone. The README at 610-615 says "Both helpers use the same atomic Method-side deopt dance … Methods whose c2i adapter cannot be recovered are skipped rather than left in an inconsistent state." — this is true of the sweep but *not* of the per-hook install, which deliberately accepts the inconsistency to keep the hook live. Decide on one policy (or document the divergence) so users know which helper to trust for production sweeps.
- `verify_hooks`' Mode 3 repair path (vmhook.hpp:8265-8273) and the install path are near-duplicates of the same write trio, and so is `try_reinstall` (vmhook.hpp:8116-8126). Three places drift together; consolidate into a single inline `apply_method_side_deopt(method*, void* i2i)` helper that returns `bool` (true if c2i was recovered) so future fixes (ARM64 ordering, stale `_code` re-read) live in one place.
- `shutdown_hooks` (vmhook.hpp:8442-8466) and `hook_handle::stop` (vmhook.hpp:8511-8517) both deliberately leave `_code` / `_from_compiled_entry` / `_from_interpreted_entry` in their install-time deopted state. The `hooked_method` struct's `original_*` fields and the struct doc at vmhook.hpp:5694-5696 still imply restoration; the dead-code improvement above would re-align the data model with the actual behaviour.
- No standalone or JVM-integration test currently exercises `deoptimize_all_jit_compiled_methods` or `deoptimize_methods_if` (greps return zero matches in `tests/` and `vmhook/src/example.cpp`). This is the highest-value test gap: the README's headline workflow has no automated coverage.
